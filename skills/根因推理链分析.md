# 根因推理链分析

## 元信息
- **版本**: 1.0
- **创建时间**: 2026-04-09
- **最后更新**: 2026-04-09
- **触发条件**: 用户提到根因分析、根本原因、为什么退货多、不良原因追溯、问题定位、归因分析、原因分析

## 描述
从客退现象出发，沿"现象→共性特征→物料/供应商追溯→根因定位→改善建议"的推理链路，进行深层根因分析。

## 知识
- 根因分析的核心是找共性：多个退货SN是否指向同一物料、同一供应商、同一批次、同一生产工厂
- 推理链路：客退现象(return_data) → SN物料追溯(sn_quality_key_material) → 供应商质量(supplier表) → 根因定位
- 数据关联键：sn_no 贯穿 return_data、sn_quality_data、sn_quality_key_material、maintain_consume_material 四张表
- supplier_name 可从 sn_quality_key_material 关联到 supplier_quality_iqc 和 supplier_performance_comparison
- material_code 可从 sn_quality_key_material 关联到 supplier_performance_comparison 进行同物料多供应商对比
- defect_material、defect_material_supplier、defect_cause 字段可能包含多个值（逗号分隔），需拆分后统计
- defect_material_batch（不良物料批次号）仅存在于 return_data 表，是追溯批次问题的关键字段
- maintain_consume_material 记录了维修实际更换的物料，是确认根因的强信号
- 当单一不良原因占比超过40%时，存在集中性不良，应重点追溯
- 当多个SN的不良物料指向同一供应商时，大概率是供应商来料问题

## 流程
1. **现象识别**：调用 `return_overview` 获取客退概况，识别TOP不良原因和TOP不良物料
2. **聚焦异常**：对占比最高的不良原因/物料，调用 `query_table` 查询 return_data 获取相关SN列表（含 defect_material_batch、defect_material_supplier 等字段）
3. **SN级追溯**：对典型SN调用 `sn_full_trace` 获取该SN使用的关键物料和供应商信息
4. **物料批次关联**：调用 `query_table` 查 sn_quality_key_material，按不良物料的供应商统计关联SN数量
5. **供应商验证**：对嫌疑供应商调用 `supplier_overview` 检查其IQC合格率和月度趋势
6. **维修物料确认**：调用 `query_table` 查 maintain_consume_material，确认维修实际更换了什么物料
7. **基线对比**：调用 `baseline_compare` 对比供应商IQC合格率与基线标准
8. **综合归因**：汇总证据链，给出根因判定和改善建议

## 输出格式
### 一、问题概述
简述客退异常现象：什么产品、退货量、主要不良表现。

### 二、不良集中度分析
展示TOP不良原因和TOP不良物料的占比，标注是否超过预警阈值（40%）。

### 三、物料追溯
用表格展示：不良物料 → 供应商 → 涉及SN数量 → 不良物料批次号（如有）

### 四、供应商质量验证
展示嫌疑供应商的IQC批次合格率、退货率，与基线对比并标注等级（优秀/合格/预警/不合格）。

### 五、维修确认
展示维修实际更换的物料统计，验证是否与追溯结果一致。

### 六、根因判定
用证据链形式输出：
```
退货现象 → 不良集中在XX物料(占比XX%) → 该物料来自供应商YY → 
供应商IQC合格率仅ZZ%(低于基线) → 批次号BBB涉及NN台 → 
根因：供应商YY的XX物料批次BBB存在来料不良
```

### 七、改善建议
给出3-5条针对性改善措施，包括：紧急措施（如暂停使用、全检）、短期措施（如供应商辅导、加严检验）、长期措施（如替代供应商开发、设计优化）。
